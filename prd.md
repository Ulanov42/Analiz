# PRD: Поиск аналогов оборудования

**Статус:** черновик · требуется решение по архитектуре (см. раздел «Открытый вопрос»)
**Владелец:** отдел закупок
**Дата:** июнь 2026

---

## 1. Проблема

Отдел закупок периодически ищет альтернативные/аналогичные предложения для оборудования, которое планируется закупить — чтобы сравнить варианты по цене, поставщику и характеристикам. Сейчас это делается вручную: сотрудник вбивает марку и модель в поисковик и сам анализирует результаты. Нужен единый инструмент, который ускоряет и стандартизирует этот процесс для всего отдела.

## 2. Цель

Веб-приложение, в котором сотрудник вводит марку и модель оборудования, а приложение само находит в интернете аналогичные модели у разных поставщиков — с ценой, регионом и ссылкой на источник — и сохраняет результат в общий список, доступный всему отделу.

## 3. Пользователи

- Сотрудники отдела закупок (несколько человек, общий доступ к одному списку).
- Без ролей/прав доступа — все участники равны, видят и могут добавлять/удалять позиции.

## 4. Ключевые сценарии использования

1. Сотрудник вводит бренд, модель, категорию и комментарий по нужному оборудованию в нижней строке формы.
2. Приложение ищет в интернете 3–5 аналогичных моделей от других производителей.
3. Для каждого аналога показываются: название, производитель, регион поставщика (Казахстан/СНГ или зарубежный), поставщик, цена (если найдена), причина соответствия, ссылка на источник.
4. Результат сохраняется в общий список — все сотрудники отдела видят одну и ту же позицию и её аналоги при открытии приложения.
5. Любой сотрудник может повторить поиск по позиции или удалить позицию из общего списка.

## 5. Функциональные требования

| № | Требование |
|---|---|
| F1 | Форма добавления оборудования всегда находится в нижней строке списка: поля «Бренд», «Модель» (обязательные), «Категория», «Комментарий/характеристики» (опциональные) |
| F2 | После добавления приложение автоматически запускает поиск аналогов в интернете |
| F3 | Результаты поиска отображаются в таблице: аналог, производитель, регион, поставщик, цена, причина соответствия, ссылка на источник |
| F4 | Список оборудования и найденных аналогов — общий для всех, кто открывает приложение (без личных аккаунтов/логина) |
| F5 | У каждой позиции виден статус: идёт поиск / готово / ошибка |
| F6 | Возможность повторить поиск по позиции или удалить её |
| F7 | Поиск должен охватывать как местных (Казахстан/СНГ), так и зарубежных поставщиков |
| F8 | Имя добавившего сотрудника фиксируется при каждой позиции |

## 6. Нефункциональные требования

- Интерфейс на русском языке.
- Работает без установки — веб-страница.
- Не требует личной регистрации/пароля для использования.

## 7. Текущая реализация и обнаруженное техническое ограничение

Прототип реализован как HTML/JS-приложение, использующее две возможности, доступные **только при открытии внутри чата Claude (как live-превью), а не при скачивании файла**:

- Прямой вызов Anthropic API с веб-поиском для подбора аналогов — без необходимости отдельного API-ключа, пока приложение открыто в превью Claude.
- Общее хранилище данных (список оборудования), синхронизированное у всех, кто открывает это превью.

**Ограничение:** при скачивании файла и открытии его как обычной HTML-страницы в браузере обе функции перестают работать — браузер блокирует прямой запрос к api.anthropic.com (ошибка `Failed to fetch`, CORS), а общее хранилище недоступно вне среды Claude.

## 8. Открытый вопрос — выбор архитектуры

Нужно выбрать один из двух путей:

**Вариант А — использовать как живое превью в чате Claude.**
- Плюсы: ничего не нужно переделывать, поиск и общий список уже работают.
- Минусы: приложение не существует как самостоятельный файл/сайт; чтобы пользоваться, нужно открывать его через этот разговор в Claude.

**Вариант Б — самостоятельный файл, открываемый в любом браузере.**
- Поиск: каждый сотрудник один раз вводит в приложении свой собственный API-ключ Anthropic (хранится только в его браузере).
- Общий список: без отдельного сервера/базы данных общий доступ для всех сотрудников реализовать нельзя — каждый браузер будет видеть только свои записи. Для настоящего общего списка потребуется внешний backend (например, общая база данных или таблица с API), что выходит за рамки простого статического файла.

Решение по выбору варианта определяет дальнейшую разработку.

## 9. Модель данных (позиция оборудования)

```json
{
  "id": "уникальный идентификатор",
  "brand": "бренд",
  "model": "модель",
  "category": "категория (опционально)",
  "notes": "комментарий/характеристики (опционально)",
  "addedBy": "имя сотрудника",
  "addedAt": "время добавления",
  "status": "searching | done | error",
  "alternatives": [
    {
      "name": "модель аналога",
      "manufacturer": "производитель",
      "match_reason": "причина соответствия",
      "region": "Казахстан/СНГ | Зарубежный",
      "supplier": "поставщик",
      "price": "цена или null",
      "source_url": "ссылка на источник или null"
    }
  ]
}
```

## 10. Вне рамок (на данном этапе)

- Личные аккаунты, роли и права доступа.
- Интеграция с внутренними системами закупок/учёта.
- Экспорт в Excel/PDF (можно добавить позже по запросу).
- Гарантия точности цен — они зависят от качества данных в открытом интернете на момент поиска.

## 11. Возможные доработки в будущем

- Экспорт списка в Excel.
- Фильтр/сортировка по статусу, категории, региону.
- История изменений по позиции (кто и когда запускал повторный поиск).
- При выборе варианта Б — отдельный backend для настоящего общего доступа без привязки к чату Claude.
